# 2026-06-17 QuickComment Logo Design Spec

## 概述

为 **QuickComment / 评论快填** 设计一个适合浏览器扩展场景使用的品牌 Logo。该 Logo 将服务于浏览器扩展图标、README 展示、GitHub 项目品牌展示以及后续 favicon 等小尺寸场景。

本次设计目标不是制作复杂品牌系统，而是先确定一个清晰、稳定、可延展的核心图形方向，并为后续使用 logoloom 生成 SVG 与品牌资产提供明确规范。

## 项目背景

QuickComment 是一个浏览器扩展，用于在博客评论区中快速填充用户信息。根据现有 README，项目重点是：

- 评论系统相关场景
- 一键快速填充
- 工具属性明显
- 面向偏技术型用户与博客评论使用场景

因此，Logo 需要同时体现“评论”和“高效工具”两层属性，而不能仅仅像一个普通聊天应用图标。

## 设计目标

本次 Logo 设计需要满足以下目标：

1. **表达评论属性**：让用户能够快速联想到评论、输入、交流语境。
2. **体现技术工具感**：避免过度社交化，更像开发者工具或浏览器插件。
3. **适配小尺寸图标**：在扩展栏、favicon 等小尺寸下仍然清晰可辨。
4. **具备较高辨识度**：通过轮廓和配色形成明确记忆点。
5. **便于导出与复用**：适合输出 SVG、PNG、多尺寸图标与单色变体。

## 已确认的设计方向

### 品牌气质
- 极客 / 技术感

### 核心意象
- 评论 / 对话气泡

### 呈现方式
- 纯图形图标

### 颜色方向
- 橙黄高辨识度

## 方案对比与决策

在讨论中已比较过以下三个方向：

### 方案 A：气泡 + 光标切角（已选定）
使用一个简洁的对话气泡作为主体，并在局部加入利落的切角结构，以传达更偏技术工具的气质。

**优点：**
- 简洁稳定
- 小尺寸识别度高
- 兼顾评论识别与工具感
- 适合浏览器扩展图标场景

**潜在风险：**
- 若切角处理过弱，技术感可能不够突出

### 方案 B：气泡 + 速度线
在气泡周围加入速度线或推进感结构，强调“快速填充”。

**未选原因：**
- 缩小时可能变杂
- 容易牺牲简洁性

### 方案 C：像素气泡 / 极客气泡
通过像素边角、数字感造型突出极客感。

**未选原因：**
- 风格更强，但通用性略弱
- 处理不好容易显得复古或不够现代

最终选择 **方案 A：气泡 + 光标切角**，作为本次 Logo 的唯一设计方向。

## 核心图形定义

Logo 为一个 **纯图形符号**，不包含项目名称文字。

### 图形组成
1. **主体结构：对话气泡**
   - 使用对话气泡作为第一识别层，保证“评论”语义直接成立。
   - 气泡不能过度圆润，应避免典型社交聊天 App 的轻松娱乐感。

2. **几何化轮廓**
   - 气泡轮廓应偏几何化，可采用圆角矩形与轻微棱角结合的方式。
   - 轮廓需要稳定、厚实，不应过细。

3. **右下角切角结构**
   - 在右下角或靠近尾部的位置加入一个干净、利落的斜切结构。
   - 此结构不是直接画成鼠标箭头，而是通过“切角”暗示点击、操作、工具属性。
   - 该切角应成为 Logo 的第二识别点。

### 图形语义
整体需要形成这样的观看顺序：
- 第一眼：这是一个与评论/对话相关的图标
- 第二眼：这不是普通聊天图标，而是一个偏技术、偏效率的工具

## 视觉风格规范

### 应坚持的风格
- 简洁
- 几何化
- 利落
- 现代
- 工具化
- 偏开发者产品气质

### 应避免的风格
- 过度圆润的社交产品气质
- 表情化、卡通化
- 复杂内部装饰
- 多余速度线
- 拟物阴影和厚重立体效果

### 线条与体量
- 优先使用较厚实、清晰的图形体块
- 不依赖细线条来表达细节
- 识别重点应来自整体轮廓，而不是内部微小元素

## 配色规范

### 主色方向
采用 **橙黄高辨识度** 路线，但要避免廉价卡通橙。

建议配色逻辑：
- **主色**：亮橙偏金黄，用于主体视觉识别
- **辅助色**：更深一点的橙色，用于层次、阴影或深色背景适配

### 配色目标
- 醒目
- 高识别度
- 保持产品感与技术感
- 在浅色与深色背景下均可稳定呈现

### 需要产出的颜色版本
1. 标准彩色版
2. 深色背景适配版
3. 纯黑版
4. 纯白版

## 使用场景要求

该 Logo 需要优先适配以下场景：

1. **浏览器扩展图标**
   - 这是最核心场景
   - 必须保证小尺寸下轮廓仍清晰

2. **README / GitHub 展示**
   - 需要在文档封面图或说明区域保持品牌感

3. **favicon / 小图标**
   - 要能被简化使用，不依赖文字

4. **后续品牌扩展**
   - 为导出 PNG、多尺寸图标、OG 图或品牌卡片保留延展空间

## 产出要求

后续使用 logoloom 实现时，至少需要产出以下资产：

### 必选
- 1 个最终版 SVG Logo
- PNG 多尺寸导出
- 单色版
- 深色背景适配版

### 推荐
- 完整 brand kit（若工具链允许）
  - 256 / 512 / 1024 PNG
  - favicon ICO
  - WebP
  - OG image
  - BRAND.md 简要品牌说明

## 验收标准

一个合格的最终 Logo 应满足：

1. 不看文字也能大致联想到评论/对话语境。
2. 与普通聊天产品图标相比，具有更明显的工具感与技术感。
3. 在小尺寸下仍然能分辨出独特轮廓。
4. 配色醒目但不过度卡通。
5. 适合作为浏览器扩展长期使用。

## 范围边界

本次设计 spec 的范围仅包括：
- Logo 方向确认
- 图形与风格约束
- 配色与输出要求

本次不包括：
- 完整品牌视觉系统
- 字体系统
- 官网或应用界面视觉规范
- 动态 Logo 动效

## 后续实施建议

进入实现阶段后，应按以下顺序推进：

1. 基于本 spec 输出 1–3 个接近最终方向的 SVG 草案
2. 从中选择最稳定的一版
3. 使用 logoloom 优化 SVG
4. 导出多尺寸品牌资产
5. 将最终 Logo 集成到 README 与项目资源中
